# AI × 人工协作流程规范（外部依赖受阻场景）

本文档定义在重构任务执行过程中，AI 遇到无法独立解决的外部依赖问题时的标准协作流程，确保过程透明、可追溯、可验证。

## 1. 适用范围

适用于以下阻塞类型（包括但不限于）：
- 用户未登录（业务系统、控制台、第三方平台）
- 权限不足（接口权限、库表权限、云资源权限、仓库权限）
- 服务未启动/不可达（Nacos、Redis、MySQL、ES、RabbitMQ、网关等）
- 配置缺失（环境变量、配置中心 dataId、密钥、证书）
- 本地文件缺失（密钥文件、授权文件、样例数据）
- 外部平台交互必须人工完成（验证码、扫码登录、审批流）

## 2. 核心原则

- **立即暂停**：一旦确认属于外部依赖阻塞，立即停止继续执行当前链路上的变更/验证动作。
- **一次说清**：明确说明“卡在哪一步、为什么 AI 无法继续、需要用户做什么”。
- **可执行指令**：给出用户可直接执行的命令/路径/操作步骤与完成标准。
- **确认后恢复**：仅在用户明确“已完成”后恢复执行，恢复时先做可达性校验再继续。
- **全程留痕**：在进度日志与交付报告中记录“阻塞-处理-恢复-结果”。

## 3. 标准流程（SOP）

### 阶段 A：阻塞识别

触发条件：
- 连续两次同类失败且错误指向外部条件（401/403/connection refused/配置占位符缺失等）
- 明确需要人工输入或平台操作（登录、审批、上传证书等）

执行动作：
- 停止当前链路后续操作
- 提取关键证据（报错摘要、模块、命令、文件路径）

### 阶段 B：阻塞通知（必须结构化）

向用户输出统一结构：
- **受阻环节**：当前执行到哪一步
- **根因判断**：为何属于外部依赖而非代码问题
- **用户操作清单**：逐条可执行步骤（命令/界面路径/配置项）
- **完成判定标准**：什么结果算“完成”
- **恢复后动作**：AI 将如何继续

### 阶段 C：用户手动处理

用户执行操作后，回复“已完成 + 关键结果”（如端口可达、登录成功、配置已写入）。

### 阶段 D：恢复执行

AI 恢复前必须先做“最小校验”：
- 服务类：端口/健康检查
- 配置类：读取配置值存在性
- 权限类：最小权限调用验证

校验通过后继续剩余任务；若失败，回到阶段 B 给出下一轮精确操作。

### 阶段 E：交付留痕

在重构报告中写清：
- 阻塞项与处理人
- 恢复校验结果
- 对范围、进度、风险的影响

## 4. 阻塞通知模板（AI对用户）

```text
【当前受阻】
- 模块/步骤：<模块名 + 当前步骤>
- 失败现象：<一句话错误摘要>

【根因判断】
- 原因：<外部依赖问题说明>
- 为什么AI无法自行处理：<权限/登录/人工审批等限制>

【请你执行以下操作】
1) <步骤1：命令或界面路径>
2) <步骤2：命令或界面路径>
3) <步骤3：命令或界面路径>

【完成标准】
- <可验证标准1>
- <可验证标准2>

【你回复后我将继续】
- 你回复“已完成”并附：<必要结果信息>
- 我会先做：<恢复前校验>
- 然后继续：<剩余任务>
```

## 5. 最终交付的强制结构

每次阶段交付必须包含以下三部分：

### 5.1 AI已完成模块 + 验收标准

- 模块清单（文件/功能点）
- 每个模块的验收标准（编译、测试、扫描、功能校验）
- 已完成证据（报告、日志、测试结果路径）

### 5.2 用户需自行测试范围 + 路径 + 预期结果

- 需要用户手测的模块范围（通常为 AI 无法模拟的登录/权限/外部平台链路）
- 测试路径（入口 URL、接口、前置账号）
- 预期结果（状态码、页面行为、日志关键字、数据变更）

### 5.3 联调/回归责任边界 + 检查清单

- AI 责任：代码改造、自动化测试、静态扫描、文档留痕
- 用户责任：账号权限、环境启动、密钥配置、业务验收
- 联调清单：接口契约、鉴权流程、异常码、性能回归、安全回归

## 6. 责任边界定义

- **AI负责**
  - 代码修改与重构实现
  - 可自动执行的测试与扫描
  - 风险识别、回滚方案、报告输出
- **用户负责**
  - 提供可用环境与访问权限
  - 执行必须人工操作步骤（登录、审批、密钥上传）
  - 对业务语义与上线策略做最终确认

## 7. 常见阻塞场景与指令示例

### 场景 1：服务未启动
- 指令：启动 Nacos/Redis/ES，返回端口检测结果
- 完成标准：`localhost:8848/6379/9200` 可达

### 场景 2：配置缺失
- 指令：在 Nacos 新增指定 dataId 与关键配置项
- 完成标准：应用启动日志不再出现占位符缺失

### 场景 3：权限不足
- 指令：授予最小必要权限（只读/写入范围明确）
- 完成标准：最小权限 API 调用通过，且无越权能力

## 8. 执行约束（本项目）

- 任何外部依赖阻塞不得“硬跳过”并伪造完成结论。
- 未通过恢复前校验，不进入下一步改造。
- 所有阻塞事件必须追加到项目日志：
  - [project-progress-log.md](file:///d:/GitProgram/gulimail/docs/refactor/logs/project-progress-log.md)
