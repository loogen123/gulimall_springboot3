# 《gulimail 重构阶段实施计划》（评审稿）

本文档将重构目标拆分为可评审、可回滚、可量化的阶段交付。阶段划分沿用你提供的 0~6 结构，并结合当前基线对“阻塞项/缺口/风险”做了必要的校正（不改变总体方向）。

## 0. 总体策略

- 原则：先“可构建、可启动、可测试、可门禁”，再做“结构性重写与领域拆分”，最后做“网关安全与可观测性完善”。
- 依赖治理优先：优先解决当前已暴露的阻塞项（测试失败、依赖缺失、配置不一致、敏感项治理），避免在领域改造阶段被反复打断。
- API 契约稳定：遵循“接口不变 → 内部重写 → 回归测试 → 性能对比”，任何契约调整必须出具 diff 与迁移策略。

## 1. 阶段 0：脚手架与基线（目标：把“评审件”变成“可门禁流水线”）

### 1.1 关键工作

- 建立并行分支：`refactor-2024Q2`
- Maven 多模块治理：统一依赖版本与插件版本；明确哪些模块纳入 reactor/CI（特别是 renren 系列）
- 引入质量门禁框架（先“可运行”，后“零警告”）
  - checkstyle / PMD / SpotBugs（至少先跑起来，并能出报告）
  - 测试与覆盖率（JaCoCo）纳入流水线
- Git Hooks 与 MR 模板：统一提交信息、格式化与基础静态检查

### 1.2 阶段门禁（DoD）

- 所有纳入 CI 的模块：`mvn -DskipTests=false test` 可执行并产出报告（允许少量已登记的已知失败但必须有阻断原因与修复计划）
- 形成“基线指标快照”：测试通过率、覆盖率、静态扫描结果（即使先是低水位，也必须可重复生成）

## 2. 阶段 1：配置与基础设施（目标：启动一致、配置可控、测试可隔离）

### 2.1 关键工作

- 配置加载策略统一
  - bootstrap → application（采用 Spring Cloud 2023 推荐加载方式）
  - 统一 Nacos namespace/group/dataId 规范；补齐 gateway/coupon namespace 缺失配置
- 日志统一：logback-spring.xml、日志格式、traceId/spanId 注入规范
- 测试基础设施：引入 Testcontainers（MySQL + Nacos + RabbitMQ + Elasticsearch 视模块需要）
  - 解决当前测试阻塞项：third-party 缺失占位符、search 依赖 ES 索引、seckill Sentinel 适配缺失
- 敏感配置治理：建立“密钥/Token/AK/SK”管理规范（脱敏、权限、审计、轮换）

### 2.2 阶段门禁（DoD）

- 所有服务：可启动、配置中心可读写、CI 绿灯
- 现有失败用例全部变为：
  - 通过；或
  - 被显式隔离到 profile，并有替代的可重复测试（不得依赖固定 IP/手工准备）

## 3. 阶段 2：公共组件与工具（目标：统一边界协议与可测试的基础库）

### 3.1 关键工作

- 统一响应体、全局异常、分页、Feign 降级、幂等令牌等横切能力
- util 包重构：删除重复轮子（与 Apache Commons 等重叠部分），补齐单元测试
- 公共组件 jar 化与示例：为各服务提供标准接入方式与示例工程/用例

### 3.2 阶段门禁（DoD）

- 组件 jar 通过质量门禁，并提供最小可运行示例与测试覆盖

## 4. 阶段 3：领域模型与数据库（目标：分层落地 + 可回滚的数据迁移）

### 4.1 关键工作

- 领域分层落地：以一个域为样板落地 API/Application/Domain/Infrastructure
- 数据库治理策略调整（建议）
  - “按聚合根重划 schema”属于高风险工作：必须拆解为可回滚的 Flyway 迁移，禁止大爆炸式替换
  - 先消除关键宽表/冗余字段的高风险点，再逐步推进范式化
- MyBatis-Plus 代码生成与规范
  - 以生成 DO/DTO/Converter 为主
  - 对“禁止手写 SQL”的执行口径建议改为：默认不手写；复杂查询允许 XML/自定义 SQL，但必须纳入评审与性能基线
- 测试：领域层单元测试 + 集成测试覆盖主流程

### 4.2 阶段门禁（DoD）

- Flyway 脚本、ER 图、测试报告齐全；评审通过方可进入下一阶段

## 5. 阶段 4：核心业务服务（目标：逐域重构，保证契约稳定与性能提升）

### 5.1 关键工作

- 五个核心域：会员、商品、订单、库存、优惠券
  - 每域遵循“四步法”：接口不变 → 内部重写 → 回归测试 → 性能对比
- 链路补齐决策点（必须在阶段开始前明确）
  - 当前“优惠券核销/积分变更”未在订单主链路接入：若纳入本次目标，需要补齐领域事件、状态机与回滚策略

### 5.2 阶段门禁（DoD）

- API 契约 diff 报告
- 压测 ≥ 基线 130%，0 阻塞 Bug

## 6. 阶段 5：接口网关与安全（目标：统一鉴权、流控与规则治理）

### 6.1 基线校正（结合现状）

- 当前仓库已存在 `gulimail-gateway`（Spring Cloud Gateway 形态）；阶段 5 建议调整为：
  - “网关升级与安全加固”而非“替换 Zuul”

### 6.2 关键工作

- 统一鉴权、JWT 刷新、防重放
- 接入 Sentinel 流控，规则持久化到 Nacos（并修复服务侧 Sentinel 适配问题）

### 6.3 阶段门禁（DoD）

- 网关源码、流控规则、渗透测试报告

## 7. 阶段 6：观测与运维（目标：可观测、可告警、可演练）

### 7.1 关键工作

- 接入 Micrometer + Prometheus + Grafana
- Actuator 健康检查：readiness/liveness 探针
- Runbook：故障演练、灰度发布、一键回滚

### 7.2 阶段门禁（DoD）

- 监控大屏、告警通道、Runbook

## 8. 里程碑与评审节奏（执行约束）

- 每阶段结束触发 Merge Request：必须 Code Review ≥ 2 人、CI 全部通过、质量门禁达标
- 每周固定评审节奏：演示阶段成果、收集阻塞列表、更新风险登记
- 全部阶段完成后：整体灰度发布演练，支持一键回滚
